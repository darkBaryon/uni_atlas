# uni_atlas 取数据策略（2026-07-04 定，2026-07-05 对账更新）

> 业务定位：**留学辅导机构**（帮在读学生过课），不是申请机构。
> 核心数据四件套：**校历考试期 / 院系结构 / 课程名单+官方链接 / module_contents 经验层**。
> 克制原则：官方维护完善的资源存链接不镜像（课程**详情**不抓）；
> 但课程**名单**（代码+名称+链接+院系归属）是核心数据，要采集入库。

## 〇之前、数据分层原则（2026-07-05 由归属问题溯源定型）

大学官网有两层，**用途不同不可混取**：
- **营销层**（招生站 study.xxx / prospective-students）：给申请者看。
  取「身份」——专业名称（学生认的名字）、对外链接、申请域字段（学费/
  雅思/截止，已降级）。**不指望它报行政结构**（归属靠页面透出=靠运气）。
- **行政层**（教务目录 DRPS / catalogue / handbook / 专业规格）：教务系统
  的公开投影。取「结构」——课程名单+代码、院系归属（索引结构本身就是
  归属）、课表、校历考试期。
两层按名称/代码 join 成完整实体。

**标准抓取路径**（2026-07-05 定，用户拍板）：
校官网 → **院系名录（行政层）→ 各系专业名单 + 课程名单**（首选，归属
随索引自带）；中央 A-Z 目录兜底保证专业枚举全量（防系页漏列——布里斯托
实测学院页不列专业）；申请页只存**身份**（名字+链接），不深解析、
**季度**刷新（quarterly，UK 6,599 张已降级，月度爬量 ~1.5万→~4千页）。
反例教训：专业从营销层取归属 → 布里斯托 37%/华威 46%/伯明翰 74% 三次翻车。

## 〇、逐校验收方法论（2026-07-05 由五校实战定型）

一次跑多校问题成堆，**一所一所验收**。每校固定三步：

1. **完整性 diff**（防爬漏）：强制重抓目录页（打脏快照 hash + 清
   `last_fetched_at`），数 `program_detail` 里 `last_fetched_at IS NULL`
   的任务；0 = 无漏页。离线 reparse 只能修「抓浅了」，验不出「漏抓了」。
2. **课程名单三问**（防源头缺登记）：官方课程目录在哪（**往往不在主站**）？
   归属的权威信号是什么（meta / 目录区块 / URL 参数）？校历版式长什么样？
3. **字段覆盖率 + 数值分布验收**：`./run.sh check <校>`；单样本会撒谎，
   聚合覆盖率也会撒谎（各校要逐个看）。

已验五校的教训分布：缺数据 ≈60% 源头没登记（目录在独立域名/门户，
浅调研碰不到）、≈30% 解析写浅（meta 没读、版式没对准）、≈10% 流程缺位；
**发现→抓取链路零漏页**（布里斯托/UCL/格拉/爱丁堡/曼大 diff 实测）。
