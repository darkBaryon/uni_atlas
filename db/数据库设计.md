# 留学院校信息库 — 数据库设计说明

> 配套文件：[schema.sql](schema.sql)（MySQL 8.0+ 可直接执行）
> 设计依据：《英国院校公开信息调研》中 10 校官网的实际信息结构。

## 选型：MySQL 8 为主，JSON 字段兜底

| 候选 | 结论 |
|---|---|
| **MySQL 8** ✅ | 核心数据（校历日期、截止日期、学费、层级关系）是强结构化的，关系型 + 外键 + 视图最适合"日常查询"；各校差异部分（奖学金、单项雅思、中介名单）用 JSON 列吸收 |
| MongoDB | 适合直接存爬下来的异构页面数据，但跨校对比查询（"所有学校 CS 硕士的截止日期排序"）写起来费劲 |
| SQLite | 如果只在本机单人用，把 schema.sql 微调后同样可用（去掉 ENUM 改 CHECK），零运维 |

原始网页快照**不进数据库**，存磁盘（`snapshots/{univ}/{category}/{date}.html`），表里只记路径和 sha256 哈希——数据库保持轻量，留痕不丢。

## 核心思想：三层数据

```
┌─ 采集层 ─────────────────────────────────────────┐
│ source_pages   每校要监控的官方页面清单（爬取任务入口）  │
│ page_snapshots 原始快照留痕（哈希变了才存新行）          │
└─────────────────┬────────────────────────────────┘
                  │ 解析 (parser)
┌─ 结构化层 ───────▼────────────────────────────────┐
│ universities ─┬─ calendar_events   校历事件          │
│               ├─ faculties         学院层级(自引用)    │
│               │    ├─ staff              师资目录      │
│               │    └─ faculty_info_items 学院动态兜底   │
│               ├─ programs ── program_details(按年份)  │
│               │    └─ program_modules ── modules 课程 │
│               │         └─ module_contents 课程内容    │
│               │           (周安排/书单/备考经验,手动为主) │
│               ├─ deadlines         截止日期            │
│               ├─ language_bands    语言分级→分数映射    │
│               ├─ china_policies    中国招生政策         │
│               └─ china_university_tiers 中国大学认可名单│
└─────────────────┬────────────────────────────────┘
                  │ upsert 时对比新旧值
┌─ 变更层 ─────────▼────────────────────────────────┐
│ change_log     字段级变更历史（学费涨了/截止提前了）      │
└──────────────────────────────────────────────────┘
```

## 关键设计决策（对应调研发现）

### 1. 专业拆成「身份 + 年度详情」
学费和截止日期**每个申请季都变**（曼大明说学费年涨最多 7%）。`programs` 存不变的身份（名称、URL、所属学院），`program_details` 每个 entry_year 一行。这样 2026 和 2027 申请季数据并存，历史可查、更新只 upsert 当季行。

### 2. 语言要求独立成 `language_bands` 表
KCL 是 Band A–E、布里斯托是 Profile A–H、UCL 是 Level 1–5，**专业页往往只标等级字母**。所以专业详情里存 `language_band` 代号，分数在 bands 表统一维护——学校调整某个 Band 的分数线时只改一行。同时 `program_details.ielts_overall` 冗余一份总分，方便直接筛选。

### 3. 截止日期单独一张表，不塞进专业详情
因为截止日期天然是多行的：分对象（international/home）、分类型（申请/押金/equal consideration）、分轮次（爱丁堡/KCL 分轮录取）、还有校级统一截止（UCAS 1 月 14 日、谢菲 PGT 总截止）。`program_id 为 NULL` 表示校级。这张表直接支撑「未来 90 天截止提醒」视图。

### 4. 校历建成「事件表」而不是宽表
各校学制不同（3 Term / 2 Semester / Teaching Block），宽表放不下。每行一个事件（教学期/考试期/假期/reading week），`calendar_track` 区分标准校历和医学院等单独校历，`academic_year` 支持多学年并存。

### 5. 中国相关信息是一等公民
- `china_policies`：校级政策（是否认高考、均分分档规则、授权中介、在华办公室）
- `china_university_tiers`：谢菲/格拉斯哥/伯明翰公开的**中国大学分级名单**逐行入库——这是给学生定位的核心数据，值得结构化

### 6. 课程信息：模块与专业多对多
「专业 (programme)」下的具体课程叫「模块 (module)」，各校模块目录公开（利兹有独立 module catalogue，其余在专业页 programme structure 里）。一个模块常被多个专业共用，所以 `modules`（模块本身：代码/学分/学期/考核方式）+ `program_modules`（关联表：必修/选修、第几学年）。模块内容也按年更新，故 `modules` 带 `entry_year`。视图 `v_program_modules` 直接回答"这个专业学什么"。

### 6b. 课程内容 `module_contents`：手动录入为主
公开渠道的深度到"完整大纲 + 阅读书单"为止（UCL 实测：讲义/试卷在 Moodle 登录墙内）。更深的内容——每周安排、讲义要点、往年考试信息、学生对难度/工作量的评价——靠手动积累，这正是辅导机构的差异化资产。设计要点：
- `content_type` 区分周主题/书单/讲义要点/往年试卷/备考经验/学生评价；
- **`source` + `verified` 是关键**：official_public（官网公开）/ official_gated（登录墙内转述，用时需谨慎）/ student（学生口述）/ agent，配合"是否已核实"，日后引用时能分清信息可信度；
- `entry_year` 留空表示常年适用；`file_path` 存本地讲义 PDF 等文件路径；
- 书单条目可由 Talis 公开 JSON 自动灌入（source='official_public'），与手动内容共存一表。

### 7. 为什么**不**按学院建表
很多信息挂在学院主页下，但这说的是**信息来源的位置**，不是数据的形状。按学院建表意味着：10 校 × 每校 3–11 个学院 = 几十上百张同构表；新增学校要改 schema；"跨校查所有商学院的金融硕士"无法用一条 SQL 表达。

正确做法：**学院是维度（一行数据），不是表**。
- 所有内容表（programs、modules、staff、faculty_info_items）挂 `faculty_id` 外键 →"某学院下的所有内容"是一个 `WHERE faculty_id=?` 查询；
- 爬取源 `source_pages` 也有 `faculty_id` →"某学院主页下要监控哪些页面"同样是查询条件；
- 学院页面上没有专门表可放的内容（新闻/研究亮点/招生宣讲），进 `faculty_info_items` 兜底表，`item_type` 区分类型。

### 8. 爬取源注册表驱动更新
`source_pages` 是唯一的任务入口：按 `crawl_freq` + `last_fetched_at` 找到期任务 → 抓取 → sha256 对比 `last_content_hash` → 没变就只更新时间戳，变了才存快照、跑解析、upsert 结构化表并写 `change_log`。`fetch_method` 标记特殊处理（华威课程列表要 JS 渲染、曼大校历是 PDF）；`status=dead/moved` 管理失效 URL（调研中已发现 5 个 404）。

## 日常更新流程

```
每日/每周 cron
  └─ SELECT * FROM source_pages WHERE status='active' AND 到期
       └─ 抓取页面 → sha256
            ├─ 哈希未变 → 更新 last_fetched_at，结束
            └─ 哈希变了 → 存 page_snapshots + 磁盘快照
                 └─ 按 category 调对应 parser
                      └─ upsert 结构化表
                           └─ 新旧值不同 → 写 change_log
```

推荐更新频率：校历/学院列表 **monthly**；申请页/截止日期 **weekly**（申请季 9–1 月可调 daily）；专业详情页 **monthly**；中国政策页 **monthly**。

## 常用查询示例

```sql
-- 1. 某校 2025/26 校历
SELECT * FROM v_calendar WHERE code='ucl' AND academic_year='2025/26';

-- 2. 未来 90 天所有截止日期（做客户提醒）
SELECT * FROM v_upcoming_deadlines;

-- 3. 跨校对比：所有 CS 类硕士的国际学费和雅思要求
SELECT * FROM v_program_full
WHERE level='PGT' AND 专业 LIKE '%Computer%' ORDER BY tuition_intl;

-- 4. 最近一周官网发生了什么变化（做"院校动态"内容）
SELECT u.name_zh, c.entity_type, c.field_name, c.old_value, c.new_value, c.detected_at
FROM change_log c JOIN universities u ON u.id=c.university_id
WHERE c.detected_at > NOW() - INTERVAL 7 DAY ORDER BY c.detected_at DESC;

-- 5. 学生本科是某中国大学，哪些学校认可、要求多少分
SELECT u.name_zh, t.tier, t.min_score, t.note
FROM china_university_tiers t JOIN universities u ON u.id=t.university_id
WHERE t.cn_univ_name LIKE '%西南大学%';

-- 6. 某专业的课程设置("这个专业学什么", 咨询高频问题)
SELECT * FROM v_program_modules
WHERE univ='sheffield' AND 专业 LIKE '%Data Science%' AND entry_year='2026';

-- 7. 某学院主页下的所有监控页面与最新动态
SELECT category, url, last_changed_at FROM source_pages WHERE faculty_id = ?;
SELECT item_type, title, published_at FROM faculty_info_items
WHERE faculty_id = ? ORDER BY published_at DESC LIMIT 20;
```

## 扩展到澳洲/港校

无需改表：`universities.country` 已含 AU/HK/SG；澳洲的 Trimester、港校的 Fall/Spring 入学用 `calendar_track` + `event_type` 表达；港校"分轮录取"用 `deadlines.round_no`；货币字段已支持 AUD/HKD。
